# Phase Plan：Phase 0 — High-Level Design Discussion

## 來源

`docs/CONTROL_PLANE.md` — Phase 0

---

## 目標

在進入 Phase 1（定義 Control Plane 模型）之前，完成對 Supabase 架構的深入分析，
確認多專案部署下的元件共用策略，並據此調整（或確認）現有的高階架構設計。

Phase 0 完成後，應產出：
1. 一份完整的 Supabase 服務架構分析文件
2. 一份元件共用可行性分析與結論
3. 若有需要，更新後的 CONTROL_PLANE.md 架構設計

---

## 進入條件

- 無（Phase 0 為起始階段）

---

## 功能拆解

| # | 功能名稱 | 設計文件路徑 | 狀態 | 說明 |
|---|----------|-------------|------|------|
| 1 | Supabase 架構分析 | `docs/designs/supabase-arch-analysis.md` | 未開始 | 盤點 docker-compose.yml 中 13 個服務的角色、依賴、資料流、狀態性與設定隔離需求 |
| 2 | 多專案共用元件分析 | `docs/designs/shared-component-analysis.md` | 未開始 | 基於功能 1 的分析，識別可共用元件，驗證共用可行性，評估資源節省與複雜度取捨 |
| 3 | High-Level Plan 調整 | `docs/designs/plan-adjustment.md` | 未開始 | 基於功能 2 的結論，決定是否調整架構圖、Runtime Adapter 介面、設定 Schema 與 Phase 1–5 內容 |

> **注意：** Phase 0 的產出為分析文件，不適用標準的設計文件範本（`_TEMPLATE.md`）。
> 各功能的文件格式可依研究主題自行定義，但須包含：分析方法、發現、結論與建議。

---

## 依賴關係

```
功能 1：Supabase 架構分析（無依賴）
  ↓
功能 2：多專案共用元件分析（依賴功能 1）
  ↓
功能 3：High-Level Plan 調整（依賴功能 2）
```

| 功能 | 依賴於 | 原因 |
|------|--------|------|
| 功能 2 | 功能 1 | 需要先理解各服務的架構角色與依賴關係，才能判斷是否可共用 |
| 功能 3 | 功能 2 | 需要先得出共用元件的結論，才能決定是否需要調整 High-Level Plan |

---

## 建議實作順序

1. **Supabase 架構分析** — 深入分析 docker-compose.yml 的 13 個服務
2. **多專案共用元件分析** — 基於架構分析，逐一評估各服務的共用可能性
3. **High-Level Plan 調整** — 綜合結論，決定架構調整方案並更新文件

---

## 退出標準

- [ ] 所有 13 個 Supabase 服務的架構角色、依賴與狀態性已完整記錄
- [ ] 每個服務的共用可行性已評估，並有明確的結論（可共用 / 不可共用 / 有條件共用）
- [ ] 若需調整架構，CONTROL_PLANE.md 已更新
- [ ] 若不需調整架構，已記錄保持現狀的理由
- [ ] Phase 0 分析結論已作為 Phase 1 的進入條件記錄

### Phase 整合驗證

Phase 0 為研究階段，無需編譯或測試。整合驗證方式為：

- 架構分析文件是否涵蓋所有 13 個服務
- 共用元件分析是否回答了所有關鍵問題
- CONTROL_PLANE.md 的一致性檢查（若有更新）

---

## 風險與待決事項

| 風險 / 待決事項 | 影響範圍 | 處理方式 |
|----------------|---------|---------|
| Studio 可能不支援 multi-project 管理 | 功能 2 結論 | 研究 Supabase Studio 原始碼或官方文件，確認其 multi-project 能力 |
| Kong 作為共用 gateway 的路由複雜度 | 功能 2、功能 3 | 評估 Kong declarative config 是否能動態路由到不同專案後端 |
| 共用元件方案可能增加架構複雜度，抵消資源節省的效益 | 功能 3 | 在功能 2 中同時評估複雜度成本，與資源節省做取捨分析 |
| 分析結論可能需要大幅修改 Phase 1–5 | 整個路線圖 | 若影響重大，在功能 3 中明確記錄變更範圍與理由 |
